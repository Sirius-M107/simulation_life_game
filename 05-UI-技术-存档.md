## 1. 开发环境

- **开发环境**：WSL2 (Ubuntu) + VS Code Windows 远程连接
- **运行环境**：纯前端网页，浏览器直接打开运行
- **无后端、无数据库、无框架、无 npm 依赖、无编译过程**

---

## 2. 技术栈

- **结构**：`index.html`
- **样式**：`style.css`（原生 CSS + Canvas 油画纹理）
- **逻辑**：`script.js`（原生 JavaScript，ES6 模块化风格、注释齐全）

### 2.1 推荐方案：原生 + Canvas

- 可实现高质量油画画布纹理（noise/grain filter）
- 卡牌和 UI 可渲染 Canvas 获得更好质感
- 粒子效果、战斗动画更流畅
- 性能优秀，主流浏览器支持好

---

## 3. 开发规范（必须遵守）

- **代码分层清晰**：数据层、游戏逻辑层、UI 渲染层、事件层分离
- **所有配置**（卡牌、怪物、事件）统一写在顶部常量区，方便后续修改拓展
- **支持本地存档**：使用 `localStorage` 保存进度、资产、卡组、解锁内容

- **整体美术风格**：中世纪油画风格（暖色调 + 画布纹理）

## 4. UI 界面布局（固定结构，油画风格）

页面分为五大固定区域，布局简洁清晰，适配卡牌战斗逻辑。

### 4.1 五大区域

1. **顶部状态栏**：显示体力值、魔力值、金币、技能、成就感、影响力、当前关卡
2. **中上战斗区**：显示当前怪物名称、血量、状态 debuff、怪物意图、**属性弱点/抗性图标**
3. **中间信息区**：战斗日志、事件提示、奖励提示
4. **底部手牌区**：横向排列当前手牌，卡牌可点击选中、使用
   - **卡牌右上角**：属性徽章（成就感/影响力）
   - **卡牌右下角**：变招/追击升级标签（如有）
5. **功能按钮区**：结束回合、查看卡组、存档、重新开局按钮

### 4.2 终结态视觉

- 怪物进入终结态时：怪物卡牌**闪烁红光** + "可终结"标签
- 卡牌在手牌中显示当前已挂载的标签（最多显示 2 个，溢出折叠为"+N"）
- 终结态被含终结属性的牌击杀 → 弹出"完美终结！"金色提示

## 5. 存档系统

基于浏览器 `localStorage` 实现本地存档，无需后端。

### 5.1 需保存数据

```js
{
  // 资源
  hp, maxHp, mp, maxMp,
  gold, skill, achievement, authority,

  // 进度
  currentLevel,        // 当前层数（1-3）
  currentNodeIndex,    // 当前节点索引
  layerBossSeed,       // 每层 BOSS 随机种子（用于恢复时保持 BOSS 选择一致）

  // 卡组（卡牌实例数组）
  deck: [
    { instanceId, id, attributes, upgrades },
    ...
  ],

  // 战斗进度（双方独立攻击回合制）
  currentEnemy,
  currentEnemyShield,  // 怪物护盾值
  playerShield,        // 玩家护盾值
  cardUsageCount,      // 变招触发器：本场战斗内每张基础卡的累计使用次数
  isFinisherState,     // 当前怪物是否处于终结态
  shield,              // 兼容字段：保留以向后兼容旧存档

  // 事件记录
  triggeredEvents: [],  // 已触发的事件 ID（防重复）
  perfectFinishers: [], // 完美终结记录

  // 伴侣系统
  partners: [
    { name, stage, satisfaction, dark }
  ],

  // 元数据
  saveVersion: '2.0',  // 已升级为 2.0（双方独立攻击回合制 + 百分比减伤）
  saveTime: timestamp
}
```

> **saveVersion 升级说明**：
> - `1.0`：旧版（点数对比制 + flat 减伤）
> - `2.0`：当前（双方独立攻击回合制 + 百分比减伤 + 机制类赐福）
> - 读取存档时若版本不匹配，建议忽略存档（`L3` 建议方案）

### 5.2 支持功能

- **自动存档**：每完成一个节点（战斗/事件/驿站/商店）自动保存
- **手动存档**：玩家主动点击"存档"按钮
- **读取存档**：从 localStorage 恢复游戏状态
- **重置开局**：清空 localStorage，重新开始

### 5.3 存档槽位

- 建议提供 3 个存档槽位（`slot1` / `slot2` / `slot3`）
- 槽位选择显示存档时间、当前层数、卡组数量等摘要

## 7. 已解决的设计问题

> 原"待明确的设计"章节中的问题已在本次重构中解决。所有冲突与不明确点详见审计报告，权威定义以本文档为准。

### 7.1 战斗公式重构（已解决 → 双方独立攻击回合制）

- ✅ **核心改动**：取消"点数对比 + 差值×2"，改为每回合双方独立攻击，1d6 作为伤害倍率（0.5x - 3x）
- ✅ **玩家攻击管线**：8 阶段结算（投骰 → 技能加成 → 变招 → 属性倍率 → 终结态 → 暴击 → 追击 → 护盾吸收）
- ✅ **怪物攻击管线**：3 阶段结算（投骰 → 基础伤害 → 百分比减伤 → 护盾吸收）
- ✅ **详细管线定义**：[02-怪物与战斗.md §3.2](./02-怪物与战斗.md) 与 §3.3

### 7.2 数值公式统一（已解决）

- ✅ **影响力威吓减伤**：明确为**百分比**减伤 `damage *= (1 - min(0.5, authority/15/100))`，上限 50%
- ✅ **技能伤害加成**：合并为 `skillBonus = 1 + skill/200`，直接作用于 `baseDamage`
- ✅ **追击独立 attributeMul**：追击段单独计算属性倍率，不与基础段叠加

### 7.3 终结态与变招·终结统一（已解决）

- ✅ **终结阈值**：统一为 20%
- ✅ **变招·终结触发器**：从 `usage_1` 改为 `hp_le_20`（条件触发，与终结态联动）

### 7.4 BOSS 数量扩展（已解决）

- ✅ **BOSS 总数**：从 4 个扩展为 **9 个**（3 层 × 每层 3 BOSS）
- ✅ **遭遇规则**：每层进入 BOSS 节点时从该层 3 选 1 随机
- ✅ **存档兼容**：通过 `layerBossSeed` 字段保持 BOSS 选择可复现

### 7.5 赐福机制化（已解决）

- ✅ **赐福类型**：从"数值加成"改为"机制类"（4 大类：抽牌/资源、减伤/反弹、状态/debuff、增益/重抽）
- ✅ **叠加规则**：机制可同时生效，不适用数值叠加（取总和/取最高）
- ✅ **调性原则**：正常组 = 保护/支援；黑化组 = 攻击/高风险
- ✅ **具体机制**：5 位同伴的具体赐福效果**待后续设计**（详见 [04-伴侣系统.md §2](./04-伴侣系统.md)）

### 7.6 委托卡主题化（已解决）

- ✅ 委托卡维持原有 HP+MP 消耗机制，本次不调整
